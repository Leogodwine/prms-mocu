FR-01: User Authentication & Authorization
FieldDescriptionRequirement No.FR-01Actor(s)Student, Supervisor, Coordinator, Head of Department (HOD), Research Office Staff, ICT Administrator, Public User (limited)Pre-Conditions• User has valid institutional credentials (students/staff) or registered account (public)
• System authentication services are operational
• Database connection is active
• User is not already logged in from another sessionInput• Login credentials (email/student ID + password)
• Registration details (name, email, role, department)
• Password reset request email
• Session timeout preference (if configurable)
• Multi-factor authentication token (if enabled)Normal Flow1. User navigates to system login page
2. User enters credentials (email/student ID and password)
3. System validates credentials against database
4. System checks user role and status (active/disabled)
5. System creates secure session with role-based permissions
6. System logs successful login with timestamp and IP address
7. System redirects user to role-specific dashboard
8. User can access authorized modules based on role
9. System displays logout option in user menu
10. User can change password from profile settingsAlternative FlowA1: Invalid credentials - System displays "Invalid email or password" message and prompts retry
A2: Account disabled - System displays "Account disabled. Contact administrator"
A3: Forgotten password - User clicks "Forgot Password", enters email, system sends reset link
A4: First-time login - System forces password change and profile completion
A5: Session expired - System redirects to login page with timeout message
A6: Multiple failed attempts - System temporarily locks account (15 minutes)
A7: Public user access - System provides read-only access to approved research without loginOutput• Successful login confirmation with dashboard access
• Session token for authenticated requests
• User role and permission set
• Login timestamp recorded in audit log
• Failed login attempt log for security monitoring
• Password reset confirmation emailPost-Conditions• User session established with appropriate permissions
• Audit trail updated with login event
• User activity can be tracked to specific user ID
• Password policies enforced (complexity, expiry)
• Session timeout configured (default 30 minutes)
FR-02: Proposal, Research reports, and Project Management
FieldDescriptionRequirement No.FR-02Actor(s)Student (Undergraduate & Postgraduate), Supervisor, Coordinator, HODPre-Conditions• User is authenticated as a student
• Student is enrolled in final year or research program
• Submission deadline has not passed (or extension granted)
• Student has not exceeded maximum submission attempts
• Active academic year/semester is configuredInput• Project title (max 500 characters)
• Research abstract (text)
• Keywords (comma-separated)
• Proposal document upload (PDF, DOC, DOCX, max 10MB)
• Research category/faculty/department selection
• Supervisor preference (if applicable)
• Funding sources (if any)
• Ethical clearance number (if obtained)Normal Flow1. Student navigates to "Create New Project" page
2. Student selects project type (Proposal/Dissertation/Thesis/Report)
3. Student completes metadata form (title, abstract, keywords)
4. Student uploads proposal document
5. System validates file type, size, and virus scan
6. Student saves as draft OR submits for review
7. For draft: System saves with status "Draft"
8. For submit: System generates unique tracking number (e.g., PRJ-2025-XXXX)
9. System assigns project to appropriate supervisor/coordinator
10. System sends email confirmation to student
11. System notifies assigned supervisor of pending review
12. Student can view submission status on dashboardAlternative FlowA1: Deadline exceeded - System blocks submission with deadline notification
A2: File too large - System rejects file with size limit message
A3: Invalid file type - System rejects with allowed formats list
A4: Missing required fields - System highlights missing fields with validation errors
A5: Duplicate title detected - System warns student and suggests modifications
A6: Supervisor not available - System queues for manual assignment by coordinator
A7: Special character issue - System sanitizes input and provides clean version
A8: Resubmission after revision - System increments version number and preserves historyOutput• Unique project tracking number
• Project record created in database
• Document stored with version control
• Confirmation email sent to student
• Notification sent to supervisor
• Project listed in student's dashboard
• Audit trail of submission createdPost-Conditions• Project available for supervisor review
• Version history maintained
• Backup of submitted document created
• Deadline compliance recorded
• Student notified of next steps
FR-03: Supervision Management
FieldDescriptionRequirement No.FR-03Actor(s)Student, Supervisor, Coordinator, HODPre-Conditions• Project has been submitted and assigned
• Valid student-supervisor relationship exists
• Supervision records are accessible
• User has appropriate permissions
• System notification services are operationalInput• Supervision meeting scheduling request
• Meeting notes and discussion summary
• Action items and deadlines
• Student progress update
• Supervisor feedback content
• Meeting type (Physical/Virtual/Written)
• Next meeting date preference
• Student progress rating (scale 1-5)Normal Flow1. User (student or supervisor) navigates to supervision module
2. System displays existing supervision history and logs
3. User requests supervision meeting with date/time proposal
4. System checks both parties' availability (if integrated)
5. System sends meeting invitation to both parties
6. Recipient accepts/declines/reschedules meeting
7. Upon acceptance, system adds to calendar
8. After meeting, supervisor logs meeting notes and action items
9. Student acknowledges meeting record
10. System updates project progress based on meeting outcomes
11. Both parties can view complete supervision historyAlternative FlowA1: No supervision history - System displays "No supervision records found" message
A2: Meeting declined - System notifies requester and suggests alternatives
A3: Missed meeting - System flags for rescheduling with reason
A4: Conflict resolution needed - Supervisor/coordinator can override scheduling
A5: Emergency supervision request - Student can flag urgent requests for immediate attention
A6: Multiple supervisors - System includes all committee members in communicationsOutput• Supervision meeting scheduled and confirmed
• Meeting notes stored and timestamped
• Action items tracked with deadlines
• Progress update reflected in project dashboard
• Email notifications sent to all parties
• Supervision log updated for audit purposesPost-Conditions• Complete supervision trail maintained
• Accountability for supervision activities established
• Student progress formally documented
• At-risk indicators updated based on meeting frequency
• Quality assurance records available for review
FR-04: Submission & Evaluation Workflow
FieldDescriptionRequirement No.FR-04Actor(s)Student, Supervisor, Coordinator, HOD, Research Office StaffPre-Conditions• Project has been approved at previous stage
• Workflow stage is properly configured
• User has appropriate authorization
• Evaluation criteria/rubric defined
• Submission deadline is active (or extension granted)Input• Document submission (draft/final)
• Self-assessment or reflection statement
• Plagiarism report (if integrated)
• Evaluation scores from supervisor
• Approval/rejection decision
• Revision instructions and requirements
• Final grade/score submissionNormal Flow1. Student submits document at current workflow stage
2. System validates submission against stage requirements
3. System assigns submission to designated reviewer
4. Reviewer (supervisor/coordinator/HOD) accesses document
5. System displays evaluation rubric for the stage
6. Reviewer completes evaluation with scores and comments
7. System aggregates scores based on rubric weightings
8. Reviewer selects decision: Approve / Revision Required / Reject
9. For Revision: System captures specific revision requirements
10. For Approve: System advances to next workflow stage
11. For Reject: System provides reason and escalation options
12. Student receives notification of decision
13. System updates project status and progression
14. Final approval (if final stage) triggers archivalAlternative FlowA1: Auto-rejection after revisions exceeded - System rejects after max revision attempts
A2: Plagiarism detected - System flags for immediate review and possible rejection
A3: Multiple reviewers required - System collects evaluations from all before final decision
A4: Grade appeal - Student can initiate appeal to HOD with justification
A5: Expedited review for deadline - System prioritises submissions approaching deadline
A6: Blind review requirement - System anonymises document before reviewer accessOutput• Submission status updated in system
• Evaluation results recorded with audit trail
• Decision notification sent to all stakeholders
• Revision requirements documented (if applicable)
• Project workflow stage advanced or looped
• Final archival record created (when complete)
• Grade published to student (if final stage)Post-Conditions• Workflow consistency maintained across projects
• Evaluation fairness and transparency ensured
• Revision history preserved for quality audit
• Completion status updated for institutional reporting
• Student notified of next steps in process

• Message content and subject

FR-06: File Management & Digital Archiving
FieldDescriptionRequirement No.FR-06Actor(s)Student, Supervisor, Coordinator, HOD, Research Office Staff, ICT Administrator, Public User (read-only)Pre-Conditions• User has appropriate upload/download permissions
• Storage space is available
• File type and size are within limits
• Antivirus scanning is operational
• Archive index is searchableInput• File upload (document, source code, data set, image)
• File metadata (title, description, version, type)
• Storage location selection (project folder, archive, public repository)
• Search query (keywords, filters, date range)
• Export format preference (PDF, DOCX, CSV, ZIP)
• Version number (for updates)
• Retention period or embargo settingsNormal Flow1. User navigates to file management interface
2. User selects upload action for specific project
3. System validates file type against allowed list
4. System scans file for viruses and malware
5. System checks file size against limits
6. System generates unique filename (timestamp + original name)
7. System stores file in secure directory structure
8. System extracts metadata (title, author, date, keywords)
9. System indexes document for full-text search
10. System creates document record in database
11. User can search archive using keywords or filters
12. System returns ranked search results
13. User can preview/download authorized files
14. System logs all access activitiesAlternative FlowA1: Duplicate file detected - System prompts user to confirm version or overwrite
A2: Virus detected - System rejects upload and alerts administrator
A3: Storage quota exceeded - System rejects upload with quota notification
A4: Corrupted file - System rejects with file corruption message
A5: Embargoed document - System restricts access until embargo date
A6: Request full-text access - System queues request for approval
A7: Bulk download - System compresses multiple files into ZIP archiveOutput• File stored securely with version tracking
• Document record created in database
• Full-text search index updated
• Access permissions applied based on user role
• Download available (if authorized)
• Audit log of upload/download activity
• Retention schedule applied to archived documentsPost-Conditions• Document integrity maintained via checksums
• Version history preserved with timestamps
• Storage usage tracked and reported
• Backup processes include new files
• Access control enforced consistently
FR-07: Supervisor Assignment & Workload Management
FieldDescriptionRequirement No.FR-07Actor(s)Coordinator, HOD, System (Automated), Supervisor (read-only)Pre-Conditions• Students are enrolled and ready for assignment
• Supervisor pool is defined with capacities
• Assignment rules/algorithm is configured
• Coordinator has assignment permissions
• Academic year/period is activeInput• Student list (CSV/Excel upload or manual entry)
• Supervisor list with capacities and expertise areas
• Assignment rules (workload balancing, expertise matching)
• Manual override instructions
• Assignment deadline
• Batch assignment trigger (automated/scheduled)
• Reassignment reason (for changes)Normal Flow1. Coordinator accesses supervisor assignment module
2. System displays current workload distribution dashboard
3. Coordinator imports/verifies student list for assignment
4. Coordinator reviews supervisor capacities and expertise
5. Coordinator selects assignment method: Automatic or Manual
6. For Automatic: System applies assignment algorithm
7. Algorithm considers workload balance (max students per supervisor)
8. Algorithm matches student research area with supervisor expertise
9. Algorithm distributes students fairly across available supervisors
10. System generates proposed assignment matrix
11. Coordinator reviews and adjusts manual overrides as needed
12. Coordinator approves final assignments
13. System notifies supervisors of new student assignments
14. System notifies students of assigned supervisors
15. Assignment record stored with timestamp and approverAlternative FlowA1: Insufficient supervisors - System alerts coordinator to add supervisors or adjust capacities
A2: Expertise mismatch - System flags mismatches for coordinator decision
A3: Student requests specific supervisor - Coordinator can honour or override
A4: Supervisor at capacity - System excludes from further assignments
A5: Reassignment needed - Coordinator can reassign with reason tracking
A6: Bulk import errors - System provides error report for correctionsOutput• Student-supervisor assignment matrix created
• Workload distribution report generated
• Email notifications sent to supervisors and students
• Assignment records stored with audit trail
• Updated dashboard reflecting new assignments
• Capacity utilization reports availablePost-Conditions• Each student has assigned supervisor
• Supervisor workloads balanced within capacities
• Assignment history tracked for analysis
• Quality of matching can be evaluated
• Adjustment requests handled systematically

• User has deadline configuration permissions
• System clock is synchronised
• Notification system is operational
• Deadline enforcement rules are definedInput• Deadline name and description
• Deadline type (Proposal/Draft/Final/Progress Report)
• Target academic year/semester
• Target audience (specific program, department, or all)

• End date and time (submission cutoff)

12. System schedules reminder notifications
13. User can monitor submission progress against deadline

A5: Missed deadline handling - System applies configured late submission policy
A6: Holiday adjustment - System automatically adjusts for institutional holidaysOutput• Deadline created and activated in system
• Notifications sent to all affected users
• Calendar entries generated for users
• Reminder schedule configured
• Deadline compliance dashboard updated
• Late submissions flagged for reviewPost-Conditions• Deadlines enforced consistently across programs
• Students receive timely reminders
• Submission windows enforced by system
• Extension requests tracked and justified
• Historical deadlines archived for reference
