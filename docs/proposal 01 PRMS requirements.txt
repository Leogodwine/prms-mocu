FUNCTIONAL REQUIREMENTS
A. User Authentication & Authorization management
* The system shall allow users to log in using a username (Registration Number and staffID) and password
* The system shall support role-based access (Coordinator, Supervisor, Project Student, Research Student, Normal Student)
* The system shall prevent unauthorized access to restricted features
* Account creation for students (temporary password via university email)
* Password reset functionality
* Verify student Registration Number against SIS
* Confirm Enrollment Status = Active
* Determine student role based on year of study:
o Eligible years ? Project/Research Student
o Other years ? Normal Student
* Import required student records from SIS into local system (read-only)
B. Project & Research Management
Coordinator Requirements
* The system shall allow the coordinator to create student accounts
* The system shall allow the coordinator to create project groups (2 or more students)
* The system shall allow the coordinator to register individual research students
* The system shall allow the coordinator to assign supervisors to groups or students
* The system shall allow the coordinator to view all groups and students
* The system shall allow the coordinator to track progress of all projects and research
* The system shall allow the coordinator to view approved groups ready for presentation
* The system shall allow the coordinator to access archived projects and research
* The system shall allow the coordinator to sees only students who are active, in eligible year for project/research, in the same department or programme
Supervisor Requirements
* The system shall allow supervisors to view assigned groups and students
* The system shall display student details (name, registration number, phone, email)
* The system shall allow supervisors to view submitted proposals, reports, demos, and source code
* The system shall allow supervisors to add comments and feedback
* The system shall allow supervisors to approve or reject project stages
* The system shall allow supervisors to digitally approve final presentation forms
* The system shall allow supervisors to track progress for each assigned project
Student Requirements
* The system shall allow students to log in using registration credentials
* The system shall allow students to view assigned supervisor details
* The system shall allow students to upload proposals, reports, demos, and source code
* The system shall allow students to download submitted and approved files
* The system shall allow students to view supervisor comments and feedback
* The system shall allow students to track project status until final approval
* The system shall allow students to view and download previous projects (read-only)
Project & research requirements
* The system shall support multiple project stages (proposal, development, demo, final)
* The system shall store all documents and source code securely
* The system shall archive completed projects for future reference
D. Notifications
* Notify students of:
o Temporary password on first registration via email and phone number
o Supervisor comments
o Approval status changes
o Notify supervisors of new submissions via email
E. File Management
* Upload and download functionality for project/research documents
* Maintain version history of submissions
* Support multiple file types (PDF, DOCX, ZIP for source code)
F.  Reporting & Tracking management
* Track project/research progress by group or individual
* View previous project/research reports
* Generate summary reports for coordinators and supervisors

Functional Requirements Specification Table
This table converts the defined use cases and functional requirements into a structured functional requirements format suitable for SRS and system design.

FR-01: Student Account Registration
FieldDescriptionRequirement No.FR-01Actor(s)Student, SystemPre-ConditionsStudent exists in SIS; Enrollment status = ActiveInputRegistration NumberNormal Flow1. Student enters registration number.
2. System validates number against SIS.
3. System retrieves student details.
4. System creates account and assigns role.
5. Temporary password is generated.OutputStudent account createdPost-ConditionsAccount exists with correct role; temporary password sent
FR-02: First Login & Password Change
FieldDescriptionRequirement No.FR-02Actor(s)StudentPre-ConditionsAccount created, temporary password issuedInput* Registration Number 
* Temporary Password
* New PasswordNormal Flow1. Student logs in with temporary password.
2. System forces password change.
3. Student sets a new password.
4. System validates and saves password.OutputPassword updatedPost-ConditionsAccount activated with permanent password
