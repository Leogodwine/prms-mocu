7. Registration, Login, and Coordinator-Filtered Student Process with SIS Integration
This section provides a full and detailed process for student registration, login, SIS data pooling, conditional role assignment, and coordinator access with filtered student view.

7.1 Key Principles
* The Student Information System (SIS) remains the single source of truth for student data.
* Student records in the proposed system are read-only.
* Registration Number (Reg No) is used as the username.
* Passwords are managed only by the proposed system.
* Only active students are allowed access.
* Student role is assigned based on year of study and eligibility for projects/research.
* Coordinators can only view students who are active, eligible for projects/research, and within their department/programme.

7.2 Student Record Pooling from SIS (Data Synchronization)
Step 1: Identify Required Student Records
Retrieve only the following fields from SIS:
* Registration Number
* Full Name
* Programme
* Department
* Year of Study
* University Email
* Phone Number
* Enrollment Status (Active / Suspended/Graduated)
Step 2: Data Retrieval Method
* Secure API connection to SIS or
* Periodic import of verified student records
Step 3: Data Storage in Proposed System
* Store retrieved records locally in the system database
* Mark all fields as read-only
* SIS passwords are never accessed or stored
Step 4: Periodic Synchronization
* Enrollment status updated periodically to reflect SIS changes
* Graduated or inactive students are automatically disabled

7.3 Student Registration Process (First-Time Access)
Step 1: Registration Number Submission
* Student enters their Registration Number into the system
Step 2: SIS Verification
* System checks if:
o Registration Number exists in SIS
o Enrollment Status = Active
? If verification fails ? registration denied
? If verification succeeds ? proceed
Step 3: Conditional Role Assignment
* Determine eligibility based on Year of Study and programme:
o Eligible year for project/research ? assign Project/Research Student role
o Other years ? assign Normal Student role
Step 4: Account Creation
* System creates user account using Reg No
* Assigns the determined role
* Sets account status to Active
Step 5: Password Generation & Delivery
* System generates random temporary password
* Password is hashed before storage
* Temporary password sent to student’s university email
Step 6: First Login & Password Change
* Student logs in with Reg No and temporary password
* System forces a password change

7.4 Normal Login Process
1. Student enters Registration Number and password
2. System verifies password hash
3. System checks enrollment status (from local database or SIS sync)
4. System applies role-based access:
o Project/Research Student ? full project/research functionality
o Normal Student ? read-only access to previous projects
5. Access granted based on role

7.5 Coordinator Access with Filtered Student View
Step 1: Login Verification
* Coordinator logs in using system credentials
* System authenticates coordinator
Step 2: Filtered Student Retrieval
* System queries local database for students meeting all conditions:
o Enrollment Status = Active
o Year of Study = Eligible for projects/research
o Department = Coordinator’s Department
o Programme = Coordinator’s Programme
Step 3: Display Filtered List
* Only students meeting all criteria are displayed
* Coordinator cannot view inactive, graduated, or ineligible students
Step 4: Coordinator Actions
* Create project groups
* Assign supervisors
* Track project/research progress
Step 5: Security & Data Integrity
* Student personal data remains read-only
* Coordinator can only use data for project/research management
* No modifications to personal or academic information are allowed
